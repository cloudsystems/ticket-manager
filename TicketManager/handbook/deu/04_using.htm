4.1 Projektverwaltung

Sobald Sie das System grundlegend konfiguriert haben und Benutzer und Gruppen hinzugefügt haben, können Sie beginnen, Projekte hinzuzufügen. Nur Administratoren oder Benutzer mit den Berechtigungen "Projekt hinzufügen"/ "Projekt ändern" können Projekte verwalten.

Hier die Bedeutung der einzelnen Felder beim Hinzufügen/Ändern eines Projekts:

  • Projektname: Wie der Name schon sagt, der Name des Projekts.
  • Bereich/Teilbereich: Sie können hier Bereiche und Bereichsverantwortliche definieren. Wenn ein Benutzer einen Fehler berichtet, kann er einen Bereich/Teilbereich aussuchen. Das System weist den Fehler dann automatisch dem Bereichsverantwortlichen zu. Ist zum Beispiel das Projekt ein Mailserver, haben Sie möglicherweise folgende Bereiche: Kernel, Webmailoberfläche, Verwaltungsoberfläche, Netzwerkprotokolle. Bei "Netzwerkprotokollen, könnten Sie hier Teilbereiche wie SMTP, POP3 und IMAP4 definieren.
  • Automatische Benachrichtigung: Wenn Benutzer Fehler melden, benachrichtigt das System automatisch die Benutzer in der Liste "Automatische Benachrichtigung". Beispielsweise kann man hier die Projektmanager hinzufügen, so dass sie stets über die Fehler des Projekts Bescheid wissen.
  • Feedback Empfänger: Wenn Kunden Fehler im Feedbacksystem melden, benachrichtigt das System automatisch die Benutzer in der Liste "Feedback Empfänger" per Mail und schickt eine Blindkopie an die Benutzer in der Liste "Automatische Benachrichtigung". "Feedback Empfänger" ist z.B. für Supportpersonal gedacht.
  • Versionsnummern-Muster: Das ist das Versionierungsmuster ihrer Software. Das System erzeugt Auswahllisten anhand dieses Musters. Für eine Ziffer verwenden sie "%", für einen Buchstaben ein "@". Ein Beispiel: Die Version ist "SDS-1-1.12" und "1.12" soll über die Auswahllisten wählbar sein, ist das Muster "SDS-1-%.%%. Wenn das Versionsnummernmuster nicht definiert ist, wird ein Eingabefeld angezeigt.
  • Zugriffsberechtigt: Konfigurieren Sie die Benutzer die auf das Projekt zugreifen dürfen. Wählen Sie die Benutzer aus und klicken Sie "Hinzufügen>>" um Benutzer hinzuzufügen und "<< Entfernen" um Benutzer zu entfernen. Mit [Strg] oder [SHIFT] können Sie eine Mehrfachauswahl treffen.

4.2 Fehlerberichtsverwaltung

Nachdem Sie ein Projekt aus der Projektliste ausgewählt haben, können Sie beginnen Fehlerberichte hinzuzufügen, zu ändern oder zu löschen. Auf der Projektseite sehen Sie alle Fehlerberichte des Projekts. Benutzer können die angezeigten  Spalten in ihren Benutzereinstellungen ändern.

Der normale Lebenszyklus eines Fehlerberichts sieht folgendermaßen aus:

  • Die Qualitätssicherung oder ein Benutzer berichtet einen Fehler. Der Status des Berichts ist "Neu".
  • Ein Manager der Forschungs-/Entwicklungsabteilung weist den Fehler dem Forschungs-/Entwicklungsteam zu.
  • Das Forschungs-/Entwicklungsteam behebt den Fehler, ändert den Status auf "Behoben - Überprüfung ausstehend" und weist ihn der Qualitätssicherung zu.
  • Die Qualitätssicherung verifiziert den Fehler und seine Behebung und setzt den Status auf " Behoben und überprüft".

Das System sendet eine Email-Benachrichtigung an den-/diejenigen in  "Zugewiesen an", den/die Bereichs-/Teilbereichsverantwortlichen, und eine Kopie an diejenigen, die in der Projektverwaltung bei "Automatische Benachrichtigung definiert sind.

In der Fehlerliste des Projekts können Sie die Fehlerberichte sortieren, filtern und durchsuchen. Sie können die Spalten die angezeigt werden, unter [System] -> [Benutzereinstellungen] ändern. Ein Klick auf die Titelleiste der Liste, erlaubt ihnen die Fehlerberichte anhand jeder einzelnen Spalte zu sortieren. Auch die Sortierfolge kann durch einen Klick in die Titelleiste geändert werden.

Um zu Suchen, geben Sie einfach das Schlüsselwort in die Suchbox ein. Sie können die Operatoren "and", "or" und "not" verwenden im erweiterte Suchabfragen durchzuführen. Zum Beispiel: Wenn sie einen Report suchen, der "FTP" und "HTTP" aber nicht "HTTPS" enthält, geben Sie in die Suchbox "FTP and HTTP not HTTPS" ein. Die Suche unterscheidet nicht zwischen Groß- und Kleinschreibung. Sie sucht in den Zusammenfassungen und den Aktivitäten im Fehlerbericht.

Um Filter zu verwenden, müssen Sie zuerst einen Filter hinzufügen. Die Filterkonfiguration befindet sich unter [System] -> [Filter verwalten]. Sie können Filter, Sortierung und Suche gleichzeitig verwenden.

Benutzer können beim Eintragen eines Fehlers Dateien hinzufügen (Screenshots, Logdateien usw.) Die Maximalgröße hängt von der Konfiguration in der php.ini ab.

4.3 Feedback System

Sie können das Feedbacksystem in [System] -> [Feedback- System] aktivieren/deaktivieren. Bitte lesen Sie Kapitel 3.4 für nähere Informationen.

Das Kundenfeedbacksystem kann völlig vom Bugracker getrennt werden. Alles was Sie tun müssen, ist das Verzeichnis "feedback_site" dahin zu verschieben, wo Sie es haben wollen. Als Beispiel: Sie verwenden Bug Tracker mit der URL http://your.domain.com/bug. Das Feedbacksystem ist standardmäßig unter http://your.domain.com/bug/feedback_site. Sie können jetzt das "feedback_site"-Verzeichnis in die gleiche Verzeichnisebene wie den Bug Tracker verschieben und den Verzeichnisnahmen in "feedback" ändern. Danach können Sie auf das Feedbacksystem über die URL http://your.domain.com/feedback zugreifen.

Darüber hinaus können Sie das Verzeichnis sogar auf einen anderen Server verschieben. Zum Beispiel können Sie den Bug Tracker im internen Netzwerk ihrer Firma betreiben und das Feedbacksystem auf einem externen Webserver. Gehen Sie sicher, das sie die Datei "feedback_site/include/config.php" angepasst haben und die richtige Datenbankinformation eingetragen ist.

Die Oberfläche des Feedbacksystems ist der von Bugtracker ziemlich ähnlich, wir haben nur versucht sie noch einfacher benutzbar zu machen. Kundenbenutzer können Fehler melden, ihre Fehlermeldungen aktualisieren und in den Fehlerberichten suchen. Wenn sie einen neuen Fehlerbericht erzeugen, sehen Sie diesen unter [Projektliste] -> [Projektname] -> [Kundenfeedback]. Hier können Sie das Kundenfeedback auch aktualisieren.

Der klassische Lebenszyklus eines Kundenfeedbacks sieht so aus:

  • Ein Kunde erstellt ein Feedback
  • Das interne Kundenservice-Personal aktualisiert das Feedback und erfragt gegebenenfalls mehr Informationen
  • Der Kunde aktualisiert das Feedback nochmals.
  • Das Kundenservice-Personal entscheidet, ob der Fehler in das interne System importiert werden soll, oder schließt das Feedback.
  • Das Kundenservice-Personal importiert das Feedback als Fehlerbericht in die interne Datenbank.
  • Interne Benutzer aktualisieren den Fehlerbericht.
  • Nachdem der interne Fehlerbericht geschlossen wird, aktualisiert das System automatisch das Feedbacksystem und schließt das Feedback.

Wenn ein Kunde ein Feedback erstellt, sendet das System eine Email an die "Feedbackempfänger" die Sie für das Project gesetzt haben, und eine Blindkopie (BCC) an die Liste der Empfänger unter "Automatischen Benachrichtigung", sowie eine Kopie (CC) an andere Kundenbenutzer der selben Kundengruppe, wenn "Benachrichtigen wenn Kollegen berichten" für den Kundenbenutzer auf "JA" gesetzt ist.

Wenn Sie das Feedback eines Kunden aktualisieren, sendet das System ein Email an den Kunden, der es gemeldet hat, und wie oben beschrieben Kopien an die weiteren Empfänger. Wenn Sie sich entscheiden, das Feedback als Fehlerbericht in die Interne Datenbank zu übernehmen, wird ihnen ein Formular für einen internen Fehlerbericht angezeigt, wo Sie noch zusätzliche Informationen eingeben können. Nachdem das Feedback importiert wurde, erzeugt das System eine Verbindung zwischen dem Feedback und dem internen Fehlerbericht. Wenn der interne Fehlerbericht geschlossen wird, schließt das System auch das Feedback.

Es gibt zwei Verwaltungsmodi für das Feedbacksystem: Anonymer Modus und Kundenmodus. Sie können alle aktiviert oder deaktiviert werden. Im Anonymen Modus können können Benutzer ihren eigenen Account auf der Anmeldeseite registrieren. Im Kundenmodus müssen Sie die Kundenbenutzer hinzufügen, bevor Sie das System benutzen können.

Wenn ein Kunde sein Passwort vergisst, sendet das System ihm/ihr ein neues Passwort zu. Das neue Passwort wird nicht in der Datenbank mit den Kundendaten gespeichert, bis sich jemand damit anmeldet. Wenn also jemand eine falsche Passwortanforderung losschickt, kann der echte Benutzer sich mit seinem alten Passwort anmelden und das neue Passwort wird fallengelassen.

Es wird ein Systemlog des Feedbacksystems geschrieben. Sie können es unter [Informationen] -> [Feedback System-Log] einsehen..

4.4 FAQ Verwaltung

Jedes Projekt hat seinen eigenen FAQ Bereich. Sie können FAQ's kategorisieren; eine FAQ kann zu vielen Kategorien gehören. Der normale Ablauf im FAQ System ist folgender:

  • Ein technischer Redakteur fügt neue FAQ-Kategorien hinzu.
  • Neue FAQ's werden hinzugefügt, und an die Entwicklungsabteilung gegeben, unm den Inhalt zu überprüfen.
  • Die Entwicklungsabteilung überprüft die FAQ und setzt den Punkt @@@"ist verifiziert" auf "JA".
  • Kunden können die Überprüfte FAQ im Feedbacksystem sehen.

Um den oben beschriebenen Ablauf durchzuführen, müssen alle beteiligten Personen die Berechtigung "FAQ verwalten" haben. Nicht überprüfte FAQ's werden im Feedback System nicht angezeigt.

4.5 Dokumentenverwaltung

Das Dokumentensystem ist entworfen, um interne Dokumente, Spezifikationen, Ankündigungen usw. zu speichern. Dokumente können durch Gruppen kategorisiert werden. Benutzern kann der Zugriff auf Dokumente verwehrt werden, die nicht ihrer Gruppe gehören.

Um Dokumente anzusehen, zu aktualisieren oder zu löschen, muß ein Benutzer die passende Berechtigung haben. Dokumentenverwaltungsrechte werden auf der Gruppenverwaltungsseite konfiguriert.

4.6 Terminplanung

Diese Funktion kann zur Verwaltung von Projektterminen verwendet werden. Mitglieder des Projektteams können hier ebenfalls Termine planen. Das Recht Termine zu sehen oder zu ändern wird gruppenweise vergeben.

Nur ein Systemadministrator oder derjenige der einen Termin erstellt hat, kann diesen ändern oder löschen.
