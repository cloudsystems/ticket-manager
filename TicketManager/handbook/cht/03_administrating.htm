3.1 基本設定

在您登入後，請選擇上方的「系統設定」，您將會看到下列畫面：

您可以在此頁面中看到所有的系統設定項目，我們先來看「基本設定」：

  • 系統名稱：本名稱會在發送電子郵件時使用並顯示於畫面最上方。
  • 日期格式：選擇系統顯示的日期格式。
  • 自動重導頁面：在您進行操作送出表單之後，如果該操作成功完成，我們會顯示一個確定的頁面。如果自動重導頁面設定「是」，則會自動重導頁面到下一頁，而無需出現確定的畫面。
  • 一般使用者登入時的認證方式：Bug Tracker 有二種使用者認證的模式 ─ 「使用 Bug Tracker 資料庫存放密碼」或「使用 IMAP 伺服器」。使用 Bug Tracker 資料庫存放密碼時，使用者的密碼會被存放在資料庫中；而使用 IMAP 伺服器則是由 IMAP 伺服器進行密碼認証。值得注意的是，當使用 IMAP 模式時，您仍需新增使用者，使用者的資訊還是存放在資料庫，只有密碼才是使用 IMAP 伺服器。
  • 寄發信件時所顯示的寄件者：這是在發送電子郵件時所要使用的寄件者。
  • 寄發信件時寄件者的電子郵件：在發送電子郵件時，我們會使用這個郵件位址。
  • 寄發信件方式：Bug Tracker 在寄送電子郵件有二種方式：「使用 PHP 內建的 mail() 函式」或是「使用 Bug Tracker 的寄信函式」。如果您的 Bug Tracker 是安裝在一台可以寄發電子郵件的 FreeBSD、Linux 伺服器上，您可以使用系統 mail 指令 (預設值)。但是如果將 Bug Tracker 安裝在 Windows 的機器上，或是您的伺服器要求身份認證才能發信，則必須使用 SMTP 協定去連接其它的郵件伺服器。
  • 允許使用者訂閱專案：如果您允許使用者訂閱專案，則當使用者更新或建立回報時，已訂閱的使用者將會收到信件通知。不過，如果您的系統中有很多使用者，則每次寄發的信件可能會很多。
  • 最多可有多少領域分類：你可以在專案中設定領域分類及擁有者。本項目是用來設定最多可以有多少個分類。
  • 每個使用者最多可有多少過濾器：每個使用者都可以有自己的過濾器，這些過濾器會被用在專案列表上。本項目即是用來限制使用者最多可以設定的過濾器數量。
  • 每個使用者最多可分享多少過濾器：使用者可以將他們所設定的過濾器分享給其它人使用，這個數字就是最多允許使用者分享的數目。
  • 最多保留多少筆系統日誌：當系統日誌超過本數目，系統會自動刪除舊的記錄。

3.2 群組

每個使用者都必須屬於一個群組。Bug Tracker 的權限控制機制是建立在群組上，您可以給予群組使用某項功能或設定某個狀態的權利。例如，你可以新增一個「市場部」的群組，並予許它管理客戶回報系統；或者，你也可以新增一個「開發部經理」的群組，並讓它擁有所有權限；而「開發部工程師」設定為不能刪除回報或刪除專案。

「可以使用的狀態」可以讓您設定某個群組可以使用的狀態。例如，你可以設定只有專案經理才可以將回報的狀態設為 "By design"，而測試人員才可以將狀態設為 "Could not reproduce"。

3.3 使用者

在使用者設定的頁面，你可以新增、刪除、編輯使用者。使用者資訊最少必須有使用者名稱、郵件、及密碼。一旦使用者名稱新增後就不能修改，而且不可以包含特殊字元，例如： \$,"*<>()\ 等。

在新增使用者之後，強烈建議您不要將使用者刪除。因為所有的回報資料中可能都會參考到使用者名稱，如果您刪除了某一個使用者，原本關於該使用者的欄位都會被清空，這將不利於您日後追蹤。如果您要刪除使用者，建議您可以直接將該使用者的帳號狀態設為「停用」。

你可以設定使用者可以存取的專案，使用者只能存取「允許使用專案」打勾的專案。您也可以在專案管理頁面中去設定可以使用專案的使用者。所以，具有編輯專案權限的人也可以修改「允許使用專案」的設定。

3.4 客戶回報系統

Bug Tracker 有一個客戶回報系統的介面，您可以修改該系統的屬性：

  • 系統名稱：本系統名稱會出現在客戶回報系統的頁面最上方及發送郵件時。您可以在這裡加上您的公司名稱。
  • 寄發信件時所顯示的寄件者：當客戶回報或是內部使用者更新回報時，系統會以此寄件者寄信給客戶及內部相關人等。
  • 寄發信件時寄件者的電子郵件：這是寄發上述信件時所使用的郵件位址。
  • 回報系統登入模式：客戶回報系統有三種登入模式 ─ 客戶模式、匿名模式、客戶及匿名模式。在客戶模式下，我們只允許已存在的客戶使用者登入；而匿名模式下，客戶可以自行註冊帳號，註皿後，我們會將它放在 Anonymous 群組中。而客戶及匿名模式則為上述二種模式的組合。如果您將登入模式設為「關閉」，則客戶回報系統將不可使用。
  • 匯入內部系統時預設回覆內容：在客戶回報後，內部具有權限管理客戶回報的使用者可以更新其回報。如果該回報經確認後匯入內部系統，我們會自動將客戶回報的狀態設為 "In Process" 並寄發信件給客戶。本項目就是寄發信件的內容。
  • 當已匯入的內部回報狀態的類型為 closed 時，要自動寫入客戶回報的訊息：在客戶的回報匯入內部系統之後，若內部對應的回報狀態的類型被設為 closed，系統會自動更新客戶回報，並寄發信件。本項目即為該信件的內容。在這個欄位中，"__STATUS__" 這個特殊字會被取代為內部系統 closed 的狀態名稱。

客戶回報系統可以完全地和 Bug Tracker 分開，您可以將 feedback_site 這個目錄搬到其它目錄或是另一台伺服器上。詳細情形請參考 4.3 節的說明。

3.5 客戶管理

如果客戶回報系統的登入模式設為「客戶模式」，則只有在客戶管理中所新增的客戶使用者才可以登入。在新增客戶使用者之前，您必須先增加客戶。所謂的「客戶」就好像是群組一樣，在新增完客戶之後，您才可以開始新增該客戶下的使用者。Anonymous 是預設的系統客戶，用來做為匿名使用者的預設群組，您無法刪除它。

您可以在「允許使用專案」項目中設定某客戶可以使用者專案。在新增、編輯客戶使用者時，您可以設定客戶使用者的屬性。和系統內部使用者一樣，我們建議您不要刪除客戶使用者，若您需要移除帳號，只要將該帳號狀態設為「關閉」。

如果客戶使用者的「接收所有同僚回報的信件附本」設為「是」，則當同屬一個客戶的使用者新增或更新回報時，該使用者也會收到系統所寄發的信件。

3.6 狀態管理

在您使用 Bug Tracker 一段時間之後，我們建議您不要再修改狀態的設定。若您真的有修改的需求，您可以在狀態管理的頁面進行修改。

我們會在每個回報資料中存入狀態的編號，如果您修改了某個狀態的名稱，則回報資料中的狀態也會跟著改變。所以如果你想要將某個狀態名稱從「Duplicated」改為「重複的」，既然它們的意思一樣，沒問題，您可以修改。但如果你想要刪除「Duplicated」並增加一個 「Pending for decision」的狀態，請不要使用編輯的戶式，而應該先將「Dulicated」刪除，再新增一個「Pending for decision」。
