4.1 專案管理

在您完成基本系統設定、新增完使用者及群組，您就可以開始新增專案了。只有 admin 或是具有管理專案權限的群組才可以新增專案。

以下為新增、編輯專案時每一個欄位的說明：

  • 專案名稱：顧名思義，即是本專案的名稱。
  • 領域分類/次分類：你可以在此設定領域及其負責人。當使用者回報時，他們可以選擇該回報是屬於那一個領域的問題，系統會依您在這裡的設定自動將該回報指派給所屬領域的負責人。例如，在一個郵件伺服器的專案中，我們可以設定領域為：Kernel、Web mail、Management UI、Network protocol 等。而在「Network protocol」這個主要領域中，我們可以再細分為 SMTP、POP3、IMAP 等次分類。
  • 自動寄發信件：當使用者回報問題時，系統會自動寄發信件給本項目中的使用者。通常我們會將專案經理加到這個列表中，如此一來，他們就可以清楚地掌控每個專案的細節。
  • 客戶回報接收者：當客戶回報問題時，系統會自動寄發信件給本項目的使用者，並將密件副本寄給「自動寄發信件」項目中的使用者。所以一般我們會將客服人員加到本項目中。
  • 版本的規則：這是您軟體專案的命名規則，系統會依照這個規則產生下拉選單。如果您要在下拉選單中使用數字，請輸入一個 % 符號來代表一個數字。如果您要在版本中使用英文字母，請使用 @ 符號來表示。例如，如果你的版本是 SDS-1-1.12，而您希望系統在 1.12 這個欄位使用下拉選單，則請在規則中輸入 SDS-1-%.%%。若「版本的規則」是空白的，則系統會顯示單純的文字方塊讓使用者輸入。
  • 允許存取的使用者：設定可以存取本專案的使用者，請先選擇左方的使用者再按「增加」來新增使用者；選擇右方的使用者再按「移除」來刪除使用者。您也可以利用鍵盤上的 Ctrl 或 Shift 以複選多個使用者。

4.2 回報管理

在進入一個專案後，您就可以開始新增、更新、刪除回報。在專案列表中，您可以看到所有使用者回報，而使用者也可以自行在偏好設定中選擇本列表中所要顯示的欄位。

正常的回報流程為：

  • 測試人員回報一個問題，並將狀態設成「New」。
  • 產品開發經理將該問題指派給工程師。
  • 工程師修復該問題，將狀態設為「Fixed - need to verify」並指派給測試人員。
  • 測試人員確認該問題已修復，並將狀態設為「Fixed and verified」。

系統會自動寄發信件給「指派給」、領域分類及次分類的負責人，並將複本寄給該專案所設定的「自動寄發信件」。

在回報列表中，您可以搜尋、過濾、排序回報，您也可以在「系統」→「偏好設定」中選擇列表中所要顯示的欄位。你可以點選列表每個欄位的表頭來切換所要排序的欄位及順序。

如果您要進行搜尋，只要在搜尋的欄位中輸入關鍵字即可。您也可以在關鍵字中使用「and」、「or」、「not」來作進階搜尋。例如，您想要找所和 FTP 及 HTTP 相關的回報，但不要包含 HTTPS，則可以輸入「FTP and HTTP not HTTPS」。關鍵字搜尋功能並不區分大小寫，我們會查找摘要及記錄這二個欄位。

如果您要使用過濾器，您必須先在「系統設定」→「設定過濾器」中進行新增設定。您也可以同時使用搜尋、過濾及排序功能。

在回報問題時，使用者可以上傳附加檔案 (例如螢幕畫面、系統記錄等)，最大上傳的檔案限制將依您 php.ini 中的設定。

4.3 客戶回報系統

您可以在「系統設定」→「回報系統」中開關客戶回報系統。請參考 3.4 節關於回報系統設定的細節。

客戶回報系統可以和 Bug Tracker 完全分開，您只要將 feedback_site 這個目錄搬到其它地方即可。例如，若你正使用 http://your.domain.com/bug 連到內部的系統，而客戶回報系統的位置將會是 http://your.domain.com/bug/feedback_site。您可以將 "feedback_site" 這個目錄搬到和 Bug Tracker 同一層的目錄下，並改名為 "feedback"。如此一來，您就可以使用 http://your.domain.com/feedback 來存取客戶回報系統了。

您也可以將客戶回報系統搬到另一台網頁伺服器。例如，您可以將 Bug Tracker 放在內部專用的伺服器上，而將客戶回報系統放在外部專用的伺服器上。只要您在搬遷後記得在 feedback_site/include/config.php 設定好資料庫的資訊即可。

客戶回報系統的介面和 Bug Tracker 很像，只是我們試著讓客戶回報系統看起來更簡單、更好使用。客戶可以新增、修改、搜尋回報。在客戶新增回報後，內部人員可以在「專案列表」→「專案名稱」→「客戶回報」中看到客戶所回報的問題。你可以在該介面中更新客戶的回報。

一般而言，一個客戶回報的處理程序為：

  • 客戶回報一個問題。
  • 內部客服人員更新回報，並要求客戶提供更多資訊。
  • 客戶再次更新回報。
  • 客服人員決定是否將此回報匯入內部，如果不匯入，就直接更新並關閉此回報。否則，
  • 匯入內部資料庫。
  • 內部使用者更新回報。
  • 在內部回報關閉後，系統會自動關閉相關的客戶回報。

當客戶回報時，系統會自動寄信給您在該專案中所設定的「客戶回報接收者」，將密件複本寄給「自動寄發信件」，並將複本寄給和該使用者同客戶的、且有設定「接收所有同僚回報的信件附本」的使用者。

當您更新客戶回報時，系統會 寄信給回報的客戶，並將副本、密件副本寄給上述的使用者。如果你選擇將客戶回報匯入內部系統，畫面將秀出一個內部回報的表單供您輸入。在客戶回報匯入內部後，系統會在外部及內部之建立連結，當內部相對應的回報關閉時，外部的客戶回報也會自動關閉。

客戶回報系統有二種使用者管理模式 ─ 客戶模式及匿名模式，您也可以同時使用二者或同時取消。在匿名模式中，使用者可以自行在客戶回報系統登入頁面中註冊帳號；而客戶模式下，只有您所新增的客戶使用者才能登入系統。

如果客戶忘記密碼，系統會自動寄出新密碼給客戶。在該客戶使用新密碼登入之前，這個密碼並未存入客戶的資料中。如果有人假冒使用者要求系統寄出新密碼，當真正的使用者要登入時，仍可以使用原本的密碼登入，而之前產生的新密碼會被移除。

系統會自動記錄客戶回報系統的使用記錄，您可以在「系統資訊」→「回報系統日誌」中查看。

4.4 常見問題管理

每個專案都有獨立的常見問題系統，您可以將常見問答分類存放，一個問答可以同時屬於多個類別。以下為一般常見問題的產生程序：

  • 技術文件人員新增新的常見問答分類。
  • 新增一個新的常見問題，並指派給工程師校對。
  • 工程師校對後，將「是否已核對」設為「是」。
  • 客戶可以在回報系統中看到己核對的常見問題。

為了讓上述會存取常見問題使用者有可以進行操作，您必須讓相關人等有權利可以管理常見問題。請注意：未核對的常見問題並不會出現在客戶回報系統中。

4.5 文件管理

文件系統可以用來存放內部文件、規格文件、公告等。您可以依群組分類文件，也可以設定拒絕不屬於該文件群組的使用者存取文件。

使用者必須有權限才可以存取、新增、更新、刪除文件。您可以在系統群組管理的頁面中設定群組的文件存取權限。

4.6 行事曆

行事曆可以用來管理您的專案行程，也可以用來做為每個專案人員的行事曆。您可以依群組設定是否可以看到行事曆或是編輯行事曆。

只有系統管理者或是行事曆資料的擁有者才可以修改或刪除行事曆資料。
