4.1 项目管理

在您完成基本系统设定、新增完使用者及群组，您就可以开始新增项目了。只有 admin 或是具有管理项目权限的群组才可以新增项目。

以下为新增、编辑项目时每一个字段的说明：

  • 专案名称：顾名思义，即是本专案的名称。
  • 领域分类/次分类：你可以在此设定领域及其负责人。当使用者回报时，他们可以选择该回报是属于那一个领域的问题，系统会依您在这里的设定自动将该回报指派给所属领域的负责人。例如，在一个邮件服务器的项目中，我们可以设定领域为：Kernel、Web mail、Management UI、Network protocol 等。而在「Network protocol」这个主要领域中，我们可以再细分为 SMTP、POP3、IMAP 等次分类。
  • 自动寄发信件：当使用者回报问题时，系统会自动寄发信件给本项目中的使用者。通常我们会将项目经理加到这个列表中，如此一来，他们就可以清楚地掌控每个项目的细节。
  • 客户回报接收者：当客户回报问题时，系统会自动寄发信件给本项目的使用者，并将密件副本寄给「自动寄发信件」项目中的使用者。所以一般我们会将客服人员加到本项目中。
  • 版本的规则：这是您软件项目的命名规则，系统会依照这个规则产生下拉选单。如果您要在下拉选单中使用数字，请输入一个 % 符号来代表一个数字。如果您要在版本中使用英文字母，请使用 @ 符号来表示。例如，如果你的版本是 SDS-1-1.12，而您希望系统在 1.12 这个字段使用下拉选单，则请在规则中输入 SDS-1-%.%%。若「版本的规则」是空白的，则系统会显示单纯的文字方块让使用者输入。
  • 允许存取的使用者：设定可以存取本项目的使用者，请先选择左方的使用者再按「增加」来新增使用者；选择右方的使用者再按「移除」来删除使用者。您也可以利用键盘上的 Ctrl 或 Shift 以复选多个使用者。

4.2 回报管理

在进入一个项目后，您就可以开始新增、更新、删除回报。在项目列表中，您可以看到所有使用者回报，而使用者也可以自行在偏好设定中选择本列表中所要显示的字段。

正常的回报流程为：

  • 测试人员回报一个问题，并将状态设成「New」。
  • 产品开发经理将该问题指派给工程师。
  • 工程师修复该问题，将状态设为「Fixed - need to verify」并指派给测试人员。
  • 测试人员确认该问题已修复，并将状态设为「Fixed and verified」。

系统会自动寄发信件给「指派给」、领域分类及次分类的负责人，并将复本寄给该项目所设定的「自动寄发信件」。

在回报列表中，您可以搜寻、过滤、排序回报，您也可以在「系统」→「偏好设定」中选择列表中所要显示的字段。你可以点选列表每个字段的表头来切换所要排序的字段及顺序。

如果您要进行搜寻，只要在搜寻的字段中输入关键词即可。您也可以在关键词中使用「and」、「or」、「not」来作进阶搜寻。例如，您想要找所和 FTP 及 HTTP 相关的回报，但不要包含 HTTPS，则可以输入「FTP and HTTP not HTTPS」。关键词搜寻功能并不区分大小写，我们会查找摘要及记录这二个字段。

如果您要使用过滤器，您必须先在「系统设定」→「设定过滤器」中进行新增设定。您也可以同时使用搜寻、过滤及排序功能。

在回报问题时，使用者可以上传附加档案 (例如屏幕画面、系统记录等)，最大上传的档案限制将依您 php.ini 中的设定。

4.3 客户回报系统

您可以在「系统设定」→「回报系统」中开关客户回报系统。请参考 3.4 节关于回报系统设定的细节。

客户回报系统可以和 Bug Tracker 完全分开，您只要将 feedback_site 这个目录搬到其它地方即可。例如，若你正使用 http://your.domain.com/bug 连到内部的系统，而客户回报系统的位置将会是 http://your.domain.com/bug/feedback_site。您可以将 "feedback_site" 这个目录搬到和 Bug Tracker 同一层的目录下，并改名为 "feedback"。如此一来，您就可以使用 http://your.domain.com/feedback 来存取客户回报系统了。

您也可以将客户回报系统搬到另一台网页服务器。例如，您可以将 Bug Tracker 放在内部专用的服务器上，而将客户回报系统放在外部专用的服务器上。只要您在搬迁后记得在 feedback_site/include/config.php 设定好数据库的信息即可。

客户回报系统的接口和 Bug Tracker 很像，只是我们试着让客户回报系统看起来更简单、更好使用。客户可以新增、修改、搜寻回报。在客户新增回报后，内部人员可以在「项目列表」→「项目名称」→「客户回报」中看到客户所回报的问题。你可以在该接口中更新客户的回报。

一般而言，一个客户回报的处理程序为：

  • 客户回报一个问题。
  • 内部客服人员更新回报，并要求客户提供更多信息。
  • 客户再次更新回报。
  • 客服人员决定是否将此回报汇入内部，如果不汇入，就直接更新并关闭此回报。否则，
  • 汇入内部数据库。
  • 内部使用者更新回报。
  • 在内部回报关闭后，系统会自动关闭相关的客户回报。

当客户回报时，系统会自动寄信给您在该项目中所设定的「客户回报接收者」，将密件复本寄给「自动寄发信件」，并将复本寄给和该使用者同客户的、且有设定「接收所有同僚回报的信件附本」的使用者。

当您更新客户回报时，系统会 寄信给回报的客户，并将副本、密件副本寄给上述的使用者。如果你选择将客户回报汇入内部系统，画面将秀出一个内部回报的窗体供您输入。在客户回报汇入内部后，系统会在外部及内部之建立连结，当内部相对应的回报关闭时，外部的客户回报也会自动关闭。

客户回报系统有二种使用者管理模式 ─ 客户模式及匿名模式，您也可以同时使用二者或同时取消。在匿名模式中，使用者可以自行在客户回报系统登入页面中注册账号；而客户模式下，只有您所新增的客户使用者才能登入系统。

如果客户忘记密码，系统会自动寄出新密码给客户。在该客户使用新密码登入之前，这个密码并未存入客户的资料中。如果有人假冒使用者要求系统寄出新密码，当真正的使用者要登入时，仍可以使用原本的密码登入，而之前产生的新密码会被移除。

系统会自动记录客户回报系统的使用记录，您可以在「系统信息」→「回报系统日志」中查看。

4.4 常见问题管理

每个项目都有独立的常见问题系统，您可以将常见问答分类存放，一个问答可以同时属于多个类别。以下为一般常见问题的产生程序：

  • 技术文件人员新增新的常见问答分类。
  • 新增一个新的常见问题，并指派给工程师校对。
  • 工程师校对后，将「是否已核对」设为「是」。
  • 客户可以在回报系统中看到己核对的常见问题。

为了让上述会存取常见问题使用者有可以进行操作，您必须让相关人等有权利可以管理常见问题。请注意：未核对的常见问题并不会出现在客户回报系统中。

4.5 文件管理

文件系统可以用来存放内部文件、规格文件、公告等。您可以依群组分类文件，也可以设定拒绝不属于该文件群组的使用者存取文件。

使用者必须有权限才可以存取、新增、更新、删除文件。您可以在系统群组管理的页面中设定群组的文件存取权限。

4.6 行事历

行事历可以用来管理您的项目行程，也可以用来做为每个项目人员的行事历。您可以依群组设定是否可以看到行事历或是编辑行事历。

只有系统管理者或是行事历数据的拥有者才可以修改或删除行事历数据。
